# Дорожня карта безпеки

Стан на 2026-07-17, після аудиту всього проєкту. Закрите — внизу, відкрите — за фазами.
Кожен пункт має «готово, коли…» — критерій, який неможливо зарахувати наполовину.

## Фаза 1 — перед відкриттям доступу з інтернету (обов'язково)

**1.1 `/media` за підписаними посиланнями.**
Зараз будь-хто з URL читає зображення без входу. План: HMAC-підпис у query (`?t=`),
UI та експорт отримують підписані URL автоматично; перехідний режим лишає старі
посилання робочими з логуванням, суворий режим вмикається змінною в `.env`.
*Готово, коли:* запит без підпису в суворому режимі отримує 403, а сторінка з
редактора artline з підписаним URL рендериться.

**1.2 Шифрування ключів провайдерів у БД (Fernet, ключ із `JWT_SECRET`).**
Зараз ключі відкритим текстом у `app_settings`, у дампах і в `./backups`.
Міграція прозора (префікс `enc:v1:`), ротація `JWT_SECRET` = ввести ключі заново.
*Готово, коли:* `pg_dump` не містить жодного `sk-`/`AIza`-префікса, і після
рестарту генерація працює без повторного введення ключів.

**1.3 Троттлінг логіна: per-IP + переживає рестарти.**
Зараз тільки per-email у пам'яті процесу — обнуляється кожним деплоєм.
Рахувати в Redis (він уже є), додати ключ по IP (`X-Real-IP` від nginx).
*Готово, коли:* 20 невдалих спроб з одного IP по різних email дають 429, і
ліміт не скидається `docker compose restart api`.

**1.4 Ліміти на дорогі ендпоінти.**
`rerun`, `translate`, `styles/generate|improve`, `probe` — без частотних лімітів
і без денної стелі витрат. Мінімум: per-user rate у Redis + глобальний денний
бюджет $ у налаштуваннях зі стопом генерацій при перевищенні.
*Готово, коли:* перевищення бюджету зупиняє нові запуски з чітким повідомленням,
а не тихо списує гроші.

## Фаза 2 — перший тиждень після відкриття

**2.1 Редагування `fallback_reason` перед видачею** — зараз сирий текст винятків
видно всім ролям. Залишити коротку причину, техдеталі — тільки в лог.

**2.2 Розмірні ліміти на решту фетчів** — `fetch_html` уже обмежений 8 МБ;
додати стелю на завантаження референсів і кадрів (30 МБ) та на ZIP-архів.

**2.3 Право власності на проєкти** (якщо команда виросте): поле `owner`
вже є — додати опційний режим «редагує лише власник або admin».

**2.4 DNS rebinding** — залишковий ризик SSRF-фікса: перевірка хопа і конект
резолвляться окремо. Закривається пінінгом IP у транспорті. Низька ймовірність,
середня складність — свідомо в фазу 2.

## Фаза 3 — гігієна та процес

**3.1 Playwright-сценарій у CI**: логін → створення проєкту → перемикання
режимів діалогу → перевірка, що URL не стерся → відкриття проєкту. Замінить
grep-smoke як головну страховку фронтенду (grep-перевірки мовчки промахуються —
ловили тричі).

**3.2 Навчання з відновлення**: `sh scripts/restore-test.sh` щомісяця; перший
запуск досі не зроблено.

**3.3 Offsite-бекапи**: TrueNAS Cloud Sync теки `./backups` у B2/S3 —
налаштовується в UI TrueNAS, код готовий.

**3.4 Дрібне:** `PROGRESS_MEMO` обмежити сотнею записів; `/health` без версії
для неавторизованих; нотатка в README, що GHCR-образи публічні і містять
промпти (свідомо, ліцензія PolyForm).

## Закрито цим аудитом (2026-07-17)

- **SSRF через редиректи** — усі зовнішні фетчі йдуть через `safe_client()`:
  кожен хоп редиректу перевіряється на публічність адреси; регресійні тести в CI.
- **Стеля розміру сторінки** — 8 МБ на `fetch_html` (покриває і probe).
- **Path traversal в успадкуванні зображень** — той самий resolve-guard, що в архіві.

## Закрито раніше (для довідки)

Відмова стартувати з дефолтними секретами; API лише на loopback; сanітизація
HTML на всіх шляхах; ключі керуються з UI тільки root-адміном і маскуються;
TLS-профіль Caddy; бекапи з перевіркою відновлюваності та алертами; заголовки
безпеки з ремонтом успадкування `add_header`; sandbox превʼю без скриптів.
